==========================================
Prover9 Proof Verification for TPTP Files
==========================================

LADR-2026-3B (March 2026)


Overview
--------

Prover9 proofs on TPTP (.p) files can be independently verified using
three methods:

  - Ivy (via prooftrans) -- Converts the proof to a detailed Ivy proof
    object where every inference step is made explicit (instantiation,
    resolution, paramodulation). The Ivy format can be checked by the
    ACL2-based Ivy proof checker.

  - GDV (Global Derivation Verifier) -- Verifies TSTP-format proofs by
    checking each derived clause against its parents using an external
    ATP (typically E prover via SystemOnTPTP).

  - tstp2ladr hints -- Converts a TSTP proof from an external prover
    (E prover, Vampire) into Prover9 hints. Prover9 re-derives the
    proof using the hints for guidance, then the re-derived proof is
    verified by Ivy. This extends Ivy verification to proofs found by
    any TSTP-capable prover.

All three methods are independent of the original proving engine and
provide strong assurance that a proof is correct.


Method 1: Ivy Verification
---------------------------

Ivy is a proof checker developed by William McCune and Olga Shumsky,
based on the ACL2 theorem prover. It verifies proofs by checking that
every inference step is a valid application of one of a small set of
primitive rules. Unlike GDV, which uses an external ATP to re-derive
each step, Ivy checks the syntactic correctness of each step directly.
This makes Ivy verification both fast and completely offline.

The Ivy workflow has two stages: first, Prover9's compact proof is
expanded into the Ivy proof object format by prooftrans; then the
proof object can be checked by the Ivy/ACL2 checker.

Step 1: Run Prover9 with -ladr_out

    bin/prover9 -ladr_out -t 60 -f problem.p > proof.out

    The -ladr_out flag produces native Prover9 output (LADR format) with
    full proof justifications. Without this flag, TPTP input automatically
    triggers TSTP output, which prooftrans cannot read.

Step 2: Convert to Ivy proof object

    bin/prooftrans ivy < proof.out > proof.ivy

    This expands every Prover9 inference step into a sequence of
    primitive Ivy operations. A single Prover9 hyper-resolution step,
    for example, becomes a chain of instantiate and resolve steps with
    explicit substitutions and literal positions.


Ivy Proof Object Format
........................

The proof object is an s-expression enclosed in parentheses. It begins
with ";; BEGINNING OF PROOF OBJECT" and ends with ";; END OF PROOF
OBJECT". Each step has the form:

    (id (rule ...) clause NIL)

The primitive rules are:

  input
    An input clause (axiom, assumption, or negated conjecture).

        (1 (input) (product (identity) v0 v0) NIL)

  instantiate parent_id substitution
    Apply a substitution to a parent clause. The substitution is a
    list of (variable . term) pairs.

        (60 (instantiate 11 ((v0 . (c1))))
            (or (not (lives (c1))) (or (= (agatha) (c1)) ...)) NIL)

  resolve parent1_id position1 parent2_id position2
    Binary resolution. Resolve a literal in parent1 against a literal
    in parent2. Positions are paths into the or-tree of a clause:
    () means the entire clause (unit), (1) means the first disjunct,
    (2) means the second disjunct, (2 1) means the first disjunct of
    the second branch, etc.

        (27 (resolve 60 (1) 12 ())
            (or (= (agatha) (c1)) (or (= (butler) (c1)) ...)) NIL)

  paramod parent1_id position1 parent2_id position2
    Paramodulation. Replace a subterm in parent2 at position2 using
    the equality in parent1 at position1.

        (53 (paramod 100 (1) 53C (3)) (product (c) (b) (a)) NIL)

  flip parent_id position
    Flip an equality literal (swap left and right sides of =).
    Position identifies which literal to flip.

        (25 (flip 24 ()) (not (= (butler) (agatha))) NIL)

  propositional parent_id
    A propositional simplification (e.g., removing a false disjunct,
    simplifying true/false, or unit deletion).

        (78 (propositional 77) false NIL)

The final step in a refutation proof derives "false":

    (78 (resolve 77 () 53 ()) false NIL)

Variables are written as v0, v1, etc. Skolem constants and functions
are written as c1, c2, f1, f2, etc. Certain symbols that are not
acceptable to ACL2 are renamed (0 --> zero_for_ivy, 1 --> one_for_ivy,
etc.).


Ivy/ACL2 Proof Checker
.......................

The Ivy proof checker is an ACL2 program that reads a proof object and
mechanically verifies each step. It checks:

  - Each "input" step corresponds to a clause in the problem
  - Each "instantiate" step correctly applies its substitution
  - Each "resolve" step is a valid binary resolution at the given
    literal positions, with a valid unifier
  - Each "paramod" step is a valid paramodulation at the given
    positions, with a valid unifier
  - Each "flip" step correctly reverses an equality
  - Each "propositional" step is a valid propositional consequence
  - The final clause is "false" (the empty clause)

The ACL2 proof of soundness establishes that if the checker accepts a
proof object, then the input clauses are genuinely unsatisfiable. This
gives the highest level of assurance: the correctness of verification
depends only on ACL2 and the small Ivy checker, not on Prover9.

Note: The Ivy checker cannot verify steps justified by "new_symbol"
(Skolem function introduction). If prooftrans ivy produces new_symbol
steps, those steps are trusted rather than verified.


Step 3: Run the Ivy proof checker

The Ivy checker is distributed as part of the ACL2 books, in the
directory books/workshops/1999/ivy. It can also be downloaded as a
standalone package.

    Download:  https://www.cs.unm.edu/~mccune/papers/ivy/
    ACL2:      Included in ACL2 under books/workshops/1999/ivy

The checker has two scripts in ivy-sources/util/:

  checker file    Check standalone proof objects (from prooftrans ivy).
                  This is the script to use with Prover9 proofs.
                  It extracts proof objects from the file, loads the
                  ACL2-verified simple-check book, and reports whether
                  each proof object is correct.

  ivy op file     Full Ivy pipeline (calls Otter/MACE, generates and
                  checks proof). Operations: prove, refute, disprove,
                  model. Used for standalone problems, not for checking
                  existing Prover9 proofs.

For checking Prover9 proofs, use "checker":

    ivy-sources/util/checker proof.ivy

The checker loads ACL2, verifies each proof object, and prints:

    There are 1 proof objects.
    The lengths of the proof objects are (44).
    The result of the check is (T).
    All proofs have been checked and are correct!

A result of (T) means the proof is correct. (NIL) means it failed.

The checker must have write permission in the directory containing
the input file (it creates a temporary .objs file).

Building Ivy (standalone package):

    tar xzf ivy-v2.tar.gz
    cd ivy
    ./Configure              # set paths to ACL2, Lisp
    cd ivy-sources
    make                     # ~90 minutes, builds ACL2 books
    cd ..

The Configure script sets the paths to the ACL2 saved image, Ivy
source directory, and Otter/MACE binaries in the checker and ivy
scripts.

Online checking (no local install required):

    https://www.cs.unm.edu/~mccune/ivy_check_prover9/

    Upload a Prover9 output file (produced with -ladr_out). The server
    runs prooftrans ivy and the Ivy checker automatically.


Complete Ivy Workflow
.....................

    # Step 1: Run Prover9 with LADR output
    bin/prover9 -ladr_out -t 60 -f problem.p > proof.out

    # Step 2: Convert to Ivy proof object
    bin/prooftrans ivy < proof.out > proof.ivy

    # Step 3: Verify with Ivy checker
    ivy-sources/util/checker proof.ivy

Reference:

Other prooftrans options that work with -ladr_out:

    bin/prooftrans parents_only < proof.out    Condensed proof
    bin/prooftrans renumber < proof.out         Renumber clause IDs
    bin/prooftrans expand < proof.out           Expand abbreviated steps
    bin/prooftrans hints < proof.out            Extract hints for reuse
    bin/prooftrans tstp < proof.out             Convert to TSTP format


Method 2: GDV Verification
---------------------------

GDV (Global Derivation Verifier) is an independent proof verification
tool from the TPTP project. It verifies TSTP-format proofs by checking
each derived formula against its parents using calls to external ATP
systems (typically E prover) on the SystemOnTPTP service at tptp.org.

There are two ways to produce the TSTP proof for GDV:

  Option A -- Convert existing LADR output with prooftrans tstp:

    If you already have a Prover9 proof generated with -ladr_out (e.g.,
    from the Ivy workflow above), convert it to TSTP format without
    re-running Prover9:

        bin/prooftrans tstp < proof.out > proof.p

    This is the recommended approach when a proof has already been
    generated, especially for proofs that took a long time to find.

  Option B -- Run Prover9 in TSTP mode directly:

    Prover9 produces TSTP output by default when reading TPTP input,
    so no special flags are needed:

        bin/prover9 -t 60 -f problem.p > proof.p

    This produces TSTP output with SZS status and SZS output delimiters.
    The proof is a CNFRefutation in TSTP format with inference() records.

Step 2: Verify with GDV

    ulimit -s 32768
    GDV -q2 -t 60 proof.p

    The ulimit is required on macOS -- GDV has deep recursion that
    overflows the default 8 MB stack. 32768 sets a 32 MB stack.

    GDV verbosity levels:
      -q0    Full output (all details)
      -q1    Results + ATP verification details
      -q2    Results only (recommended)
      -q3    SZS status line only

    GDV checks:
      - Derivation has unique formula names
      - Derivation is acyclic
      - Derivation looks like a refutation
      - Negated conjecture is a leaf (for FOF problems)
      - Assumptions are propagated and discharged
      - Leaf axioms are satisfiable (for FOF problems)
      - Each derived formula follows from its parents (via ATP call)

    A successful verification prints:

      SUCCESS: Derived formulae are verified
      % SZS status Verified


Example: Full GDV Workflow
..........................

    $ bin/prover9 -t 60 -f PUZ001+1.p > puz001_proof.p

    $ ulimit -s 32768
    $ /path/to/GDV -q2 -t 60 puz001_proof.p
    SUCCESS: Derivation has unique formula names
    SUCCESS: All derived formulae have parents and inference information
    SUCCESS: Derivation is acyclic
    SUCCESS: Derivation looks like a refutation
    SUCCESS: Negated conjecture c_26 is a leaf or CTH from a conjecture
    SUCCESS: Assumptions are propagated
    SUCCESS: Assumptions are discharged
    SUCCESS: Leaf axiom(_like) formulae are satisfiable
    SUCCESS: c_25 is a thm of c_24
    SUCCESS: c_27 is a thm of c_11 c_12
    ...
    SUCCESS: c_53 is a thm of c_18 c_50 c_26
    SUCCESS: Derived formulae are verified
    % SZS status Verified


Method 3: Verifying External Prover Proofs (tstp2ladr Hints)
-------------------------------------------------------------

E prover and Vampire can find proofs that Prover9 cannot, and vice
versa. The tstp2ladr hints pipeline lets you verify proofs from any
TSTP-capable prover using Ivy, by having Prover9 re-derive the proof
with the external proof as guidance.

The idea: extract the clauses from the external proof, feed them to
Prover9 as "hints" (preferred clauses), and let Prover9 find its own
proof guided by the external one. Prover9's proof is then verified by
Ivy in the usual way. If Ivy accepts, the result is independently
verified regardless of the original prover.


The tstp2ladr Tool
...................

tstp2ladr reads a TSTP proof from stdin (or a file) and converts it
to LADR format or to Prover9 hints input.

    Usage: tstp2ladr [options] [-f <file>]

    Options:
      -ladr      Output LADR-format proof (default)
      -hints     Output Prover9 input with proof clauses as hints
      -p <file>  Use original TPTP problem for sos/goals (hints mode)
      -v         Verbose: print reconstruction details to stderr
      -f <file>  Read from file instead of stdin

In -hints mode, the output contains three sections that are piped
directly into Prover9:

    formulas(sos).        % axioms and negated conjectures
    ...
    end_of_list.

    formulas(goals).      % positive conjectures (omitted if none)
    ...
    end_of_list.

    formulas(hints).      % all derived clauses from external proof
    ...
    end_of_list.


The -p Option
..............

External provers clausify FOF axioms differently than Prover9. Without
-p, Prover9 must re-derive from the external prover's clausified
axioms, which sometimes fails because the clausification diverged.

With -p, tstp2ladr reads the original TPTP problem file for sos/goals
instead of extracting them from the proof. Prover9 then clausifies the
original axioms its own way and uses the external proof only as hints.
This is more robust and is the recommended approach.


E Prover Workflow
..................

Step 1: Run E prover with proof output

    eprover --cpu-limit=60 --tstp-format -s --proof-object problem.p \
        | tstp2ladr -hints -p problem.p > hints.in

    E prover flags:
      --cpu-limit=N        Time limit in seconds (not -t, which sets
                           term ordering in E)
      --tstp-format        Output in TSTP format
      -s                   Print SZS status line
      --proof-object       Include proof object in output

Step 2: Re-derive with Prover9

    bin/prover9 -ladr_out -t 60 -f problem.p hints.in > proof.out

    Prover9 reads the original problem (-f problem.p), reads the hints
    from hints.in, and searches for a proof. The hints guide Prover9
    toward the same proof E found, typically finishing in under a second.

Step 3: Convert to Ivy proof object

    bin/prooftrans ivy < proof.out > proof.ivy

Step 4: Verify with Ivy checker

    ivy-sources/util/checker proof.ivy


Vampire Workflow
.................

Step 1: Run Vampire with proof output

    vampire --proof tptp -t 60 problem.p \
        | tstp2ladr -hints -p problem.p > hints.in

    Vampire flags:
      --proof tptp         Output TSTP-format proof
      -t N                 Time limit in seconds

    Vampire's AVATAR splitting is handled automatically. AVATAR creates
    splitting predicates (spl0_1, spl0_2, etc.) that flow through hints
    as regular formulas. Prover9 treats them as uninterpreted constants.
    There is no need to disable AVATAR (--avatar off).

Steps 2-4 are the same as the E prover workflow above.


Complete Example: E Prover
...........................

    $ eprover --cpu-limit=10 --tstp-format -s --proof-object \
          PUZ001-1.p | tstp2ladr -hints -p PUZ001-1.p > hints.in

    $ bin/prover9 -ladr_out -t 30 -f PUZ001-1.p hints.in > proof.out

    $ bin/prooftrans ivy < proof.out > proof.ivy

    $ ivy-sources/util/checker proof.ivy

    There are 1 proof objects.
    The lengths of the proof objects are (...).
    The result of the check is (T).

When to Use -p
...............

Always use -p unless you have a specific reason not to. Without -p,
re-derivation fails on some problems where the external prover's
clausification diverges from Prover9's. In testing:

  - Without -p: 16/17 problems re-derived successfully (94%)
  - With -p:    16/17 problems re-derived successfully (94%)

For E prover specifically, -p fixed 6 additional problems where
clausification diverged. There are no known regressions from using -p.


When Re-derivation Fails
..........................

If Prover9 times out (cannot re-derive the proof using hints), try:

  1. Increase the time limit (-t 120 or higher)
  2. Use -p if not already using it
  3. Try the other external prover (Vampire instead of E, or vice versa)
  4. Skip Ivy verification and use GDV on the external prover's TSTP
     output directly (GDV does not require re-derivation)

Re-derivation can fail when the external proof uses inference rules or
strategies that have no close analog in Prover9, making it difficult
for Prover9 to reconstruct the proof even with hints.

Notes and Limitations
---------------------

1. FOF vs CNF problems

   GDV works best with FOF (first-order formula) problems. For CNF
   problems, GDV may fail to verify individual inference steps because
   it uses an external ATP to re-derive each step, and some steps
   (particularly multi-nucleus hyper-resolution) can be difficult for
   the verifying ATP to reconstruct within its time limit.

   Ivy does not have this limitation -- it checks syntactic correctness
   of each step directly and works equally well on FOF and CNF problems.

2. The -ladr_out flag enables all prooftrans modes

   - Use -ladr_out to get LADR-format output that prooftrans can read
   - prooftrans tstp converts LADR output to TSTP for GDV verification
   - prooftrans ivy converts LADR output to Ivy proof objects
   - Without -ladr_out, TPTP input produces TSTP output directly
     (suitable for GDV but not for prooftrans ivy/hints/parents_only)

3. GDV requires internet access

   GDV calls remote ATP systems on tptp.org (SystemOnTPTP) to verify
   each inference step. It will not work offline.

4. Ivy works offline

   The prooftrans ivy conversion and the Ivy/ACL2 checker both run
   entirely locally. No network access is needed.

5. Combining methods

   For maximum assurance, verify the same proof with both Ivy and GDV.
   Run Prover9 once with -ladr_out, then use prooftrans for both:

       bin/prover9 -ladr_out -t 60 -f problem.p > proof.out
       bin/prooftrans ivy  < proof.out > proof.ivy    # for Ivy
       bin/prooftrans tstp < proof.out > proof.p      # for GDV

   There is no need to run Prover9 twice.

6. tstp2ladr is a separate tool

   tstp2ladr is not part of the LADR distribution. It is a standalone
   utility for converting TSTP proofs. It can be found at:

       https://github.com/...   (TODO: add URL when published)

   The -hints mode requires Prover9 and prooftrans from LADR-2026-3B
   or later. Earlier versions of Prover9 do not read hints from
   additional input files on the command line.

7. Verification trust chain

   The trust chain for each method:

   Method 1 (Ivy):    Prover9 --> prooftrans --> ACL2 checker
   Method 2 (GDV):    Prover9 --> GDV --> E prover (SystemOnTPTP)
   Method 3 (Hints):  External ATP --> tstp2ladr --> Prover9 -->
                       prooftrans --> ACL2 checker

   In Method 3, the external ATP is not trusted -- its proof is only
   used as guidance. The verified proof is Prover9's independent
   re-derivation, checked by the same Ivy/ACL2 chain as Method 1.
