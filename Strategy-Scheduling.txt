=============================================
Strategy Scheduling for Prover9
=============================================

LADR-2026-3E (March 2026)


Overview
--------

Strategy scheduling runs multiple Prover9 search strategies concurrently
using a sliding-window scheduler and takes the first proof found. Each
strategy uses different search parameters (ordering, weight limits,
given-clause selection, demodulation, SInE) drawn from a machine-learned
portfolio of 139 strategies.

The scheduler is invoked with -cores N, which forks N concurrent child
processes. A parent process manages the children using sleep/wake
signals: when a child's time slice expires, it is suspended (SIGSTOP)
and a new strategy is spawned or a previously suspended child is resumed
(SIGCONT). This sliding-window approach lets many strategies share a
small number of CPU cores.

Strategy scheduling is designed for TPTP competition use. It is enabled
by -cores N or -comp N and is completely inert in LADR mode.

Added in LADR-2026-2Z. Rewritten as the sliding-window scheduler in
LADR-2026-3C.


1. Command-Line Flags
======================

-cores <n>

    Enable the sliding-window scheduler with N concurrent children
    (1 to 64). Each child runs a different strategy from the ML
    portfolio. Auto-enables TPTP mode features (SInE, multi-ordering
    trial, fast predicate elimination).

    Example: prover9 -cores 4 -t 300 -f problem.p

-comp <n>

    Competition mode. Shorthand for -cores <cpus> -fastPE -t <n>.
    Uses all available CPUs and sets a time limit of n seconds.

    Example: prover9 -comp 60 problem.p

-strategy <n>

    Force single strategy index N from the ML portfolio (0 to 95).
    Runs a deterministic single-core search with that strategy's
    parameters. No forking, no scheduling. Overrides -cores.

    Example: prover9 -strategy 5 -t 60 -f problem.p

-strategies <n>

    Override Phase 1 breadth cap. Limits how many strategies are
    tried in Phase 1 of the sliding-window scheduler. Clamped to
    [1, 139]. See "Two-Phase Scheduling" below.


2. ML Strategy Portfolio
=========================

The portfolio contains 139 strategies: 138 specialist strategies trained
on problems that auto_default fails on, plus auto_default itself (the
standard Prover9 automatic mode with no parameter overrides).

Each specialist strategy specifies values for up to 27 search parameters:

  - Term ordering (LPO, RPO, KBO)
  - Given-clause selection (age_part, weight_part)
  - Weight limits (max_weight)
  - SInE parameters (tolerance, depth, max_axioms, weight)
  - Demodulation (back_demod, factor, unit_deletion)
  - Resolution modes (ordered_res, ordered_para, para_units_only)
  - Penalty functions (literal_selection, nest_penalty, depth_penalty)

A parameter value of -1 means "keep Prover9's default". auto_default
has all parameters set to -1.

The portfolio is defined in strategy_data.h, generated by the specialist
training pipeline (greedy set cover over a 1000-strategy grid evaluated
on hard TPTP problems).


3. Decision Tree Classifier
=============================

A decision tree classifies each input problem into a leaf node based on
syntactic features (symbol counts, clause counts, formula structure).
Each leaf node has a ranking of all 139 strategies, ordered by how often
each strategy solved problems at that leaf during training.

The decision tree and per-leaf rankings are defined in dtree_data.h,
generated by build_tree.py. The classifier runs in microseconds and
adds no measurable overhead.

When -cores N is used, the scheduler uses the per-leaf ranking to
determine which strategies to try and in what order. auto_default is
always tried first (slot 0), regardless of the ranking.


4. Two-Phase Scheduling
=========================

The scheduler operates in two phases:


Phase 1: Breadth
.................

Try as many different strategies as possible. Each child gets a fixed
time slice, and when it expires, the child is suspended and a new
strategy is spawned.

  - Number of strategies: phase1_limit = min(num_strats, T*N / (2*S))
    where T = total time, N = cores, S = minimum slice (8 seconds).
    This allocates roughly half the total budget to breadth.
  - Per-child slice: max(8, T*N / (2*phase1_limit)) seconds.
  - If phase1_limit <= N, all Phase 1 children get the full time T
    (no suspension needed).
  - The -strategies flag overrides phase1_limit.

Phase 1 ends when all phase1_limit strategies have been started.


Phase 2: Depth
...............

Resume the most promising suspended children using priority scores.
Children that made the most progress in Phase 1 are resumed first.

Priority scoring:

  - Children still in preprocessing (stage < SEARCHING): score 0.
    These are stuck in predicate elimination or clausification and
    are low priority.
  - Children that reached the main search loop: score = 1000 +
    delta*100 + total_given, where delta = given clauses processed
    since last suspend. Higher scores mean more search progress.
  - auto_default (slot 0): bonus +10000 when searching. Always
    resumed first if it has reached the search loop.

Phase 2 continues until the time limit expires, cycling through
suspended children in priority order.


5. auto_default (Slot 0)
=========================

auto_default always occupies slot 0 and receives special treatment:

  - It is the first strategy started.
  - It is never suspended. While other children cycle through
    sleep/wake, auto_default runs continuously on its dedicated core.
  - In Phase 2, it receives the highest priority bonus (+10000)
    if it has reached the main search loop.
  - It uses Prover9's standard automatic mode with no parameter
    overrides, which is the strongest single strategy.


6. Two Fork Modes
==================

The scheduler uses two different fork points depending on the number
of axioms in the input.


Fork-Before-SInE (>5000 axioms)
.................................

For problems with more than 5000 axioms, the scheduler forks BEFORE
SInE premise selection and clausification. Each child gets its own
SInE parameters and independently selects a different subset of
axioms.

  Phase 1: Scan input (shared, fast text scan)
      |
      +-- fork N children
            |
            +-- Each child:
                  Apply per-child SInE parameters
                  Parse and clausify (with own SInE filter)
                  Run search with per-child options
            |
            +-- Parent:
                  Manage sleep/wake via SIGSTOP/SIGCONT
                  Print winner's output

This mode provides diversity at BOTH the premise selection level and
the search strategy level. The scan data is copy-on-write shared
after fork, so memory overhead is minimal.


Fork-After-Clausification (<=5000 axioms)
..........................................

For problems with 5000 or fewer axioms, the scheduler forks AFTER
full preprocessing (SInE, parsing, clausification). All children
share the same clause set and differ only in search parameters.

  Full preprocessing (shared: scan, SInE, parse, clausify)
      |
      +-- fork N children
            |
            +-- Each child:
                  Apply per-child search parameters
                  Run search
            |
            +-- Parent:
                  Manage sleep/wake
                  Print winner's output

This avoids redundant preprocessing (clausification is expensive for
FOF problems) and is appropriate when the axiom set is small enough
that SInE diversity is not critical.


7. Shared-Memory Progress Hints
=================================

Children communicate their progress to the parent via shared memory
(mmap MAP_SHARED). Each child writes to a dedicated 128-byte slot
containing:

  - stage: current processing phase (INIT through DONE)
  - given: total given clauses processed
  - kept: total kept clauses
  - sos_size: current SOS list size
  - usable_size: current usable list size
  - megs_used: current memory consumption

The parent reads these slots to make Phase 2 priority decisions and
to detect children that are stuck in preprocessing or consuming too
much memory.


8. Memory Ejection
===================

When max_megs is set to 8192 or less (competition mode), the parent
enforces a per-child memory budget:

  budget = max_megs / max(num_strats, N)

Children whose megs_used exceeds the budget are killed to prevent a
single child from consuming all available memory.


9. Winner Selection
====================

The parent process waits for child exits. The first child to exit
with a proof-found status (exit code 0) is declared the winner.

When a winner is found:

  1. All other children are woken (SIGCONT) then killed (SIGTERM).
  2. The winner's proof output (saved to a temp file) is printed
     to stdout.
  3. The parent exits with the winner's exit code.

If no child finds a proof, the parent reports the best exit status
from all children:

  - SOS empty / max_given / max_kept --> GaveUp
  - max_seconds exceeded --> Timeout
  - max_megs exceeded --> MemoryOut


10. Diagnostic Output
======================

In debug builds (compiled with DEBUG=1), the scheduler prints
diagnostic information to stderr. Release builds produce clean
stderr with no scheduler diagnostics.

Debug-mode diagnostics include:

  - Strategy slot assignments and per-child parameters.
  - Suspend/resume events with progress snapshots (stage, given,
    kept, megs).
  - Phase 1/Phase 2 transitions.
  - Winner announcement and exit status.


11. Examples
=============

Example 1: 4-core scheduling, 5-minute timeout

    prover9 -cores 4 -t 300 -f problem.p

  Four concurrent children, sliding window over 139 strategies.
  Phase 1 tries as many strategies as fit in ~150 seconds, then
  Phase 2 resumes the most promising.

Example 2: Competition mode, 60 seconds

    prover9 -comp 60 problem.p

  Uses all available CPUs, fast predicate elimination, 60-second
  time limit.

Example 3: Force a specific strategy

    prover9 -strategy 5 -t 60 -f problem.p

  Runs strategy index 5 from the portfolio as a single-core
  deterministic search. No forking.

Example 4: Override Phase 1 breadth

    prover9 -cores 4 -strategies 20 -t 300 -f problem.p

  Limits Phase 1 to 20 strategies instead of the automatic cap.

Example 5: Disable scheduling (single search)

    prover9 -t 60 -f problem.p

  Without -cores or -comp, TPTP mode runs a single deterministic
  search with auto_default parameters. No forking.


12. Notes and Limitations
==========================

1. CPU usage

   With -cores N, N child processes run concurrently. On a machine
   with N+ CPU cores, each child gets a full core. On machines with
   fewer cores, children share CPU time but still get the full
   wall-clock budget.

2. Memory usage

   Each child gets its own memory space. The max_megs limit applies
   per child. For large inputs, N children each using 1-2 GB means
   N * 1-2 GB total. Memory ejection prevents runaway usage in
   competition mode.

3. Interaction with manual options

   User-specified options (via magic comments or command-line) are
   applied before strategy overrides. Strategy children then modify
   specific parameters. User settings for parameters not touched by
   the strategy are preserved.

4. Temp files

   Each child writes output to a temp file. These are cleaned up in
   all exit paths (proof, timeout, failure).

5. No effect in LADR mode

   The -cores and -comp flags are only meaningful with TPTP input.
   LADR-mode searches are completely unaffected.

6. Checkpointing

   Checkpointing is automatically disabled in strategy children
   (each child sets checkpoint_minutes=-1) to avoid checkpoint file
   collisions.
