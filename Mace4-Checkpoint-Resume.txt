====================================
Mace4 Checkpoint/Resume User Guide
====================================

LADR-2026-3E (March 2026)


Overview
--------

Mace4 can save its search state to disk (checkpoint) and later resume
from that saved state. This is useful for:

  - Long-running model searches where you want insurance against crashes
    or machine reboots
  - Splitting a long run across multiple sessions
  - Examining intermediate search state
  - Moving a run to a different machine

The checkpoint captures the DPLL search decision trail -- the sequence
of cell assignments that defines the current position in the search
tree. When resumed, Mace4 rebuilds all internal structures from the
input and replays the decision trail, then continues searching from
exactly where it left off. Model counts are accumulated across
checkpoint/resume boundaries.

Checkpointing works in both LADR mode and TPTP mode.


How It Works
------------

Unlike Prover9 (which saves clause lists and index state), Mace4 uses
a lightweight decision-trail approach. The checkpoint saves only:

  - The current domain size
  - The total number of models found so far
  - The decision trail (which cell was assigned which value at each depth)
  - The input clauses (as a self-contained LADR file)

On resume, Mace4 re-reads the input, rebuilds the cell table and ground
clauses for the saved domain size (this is deterministic), then replays
the saved assignments via assign_and_propagate(). This reconstructs the
exact same search state without needing to serialize internal pointers or
data structures.

Typical checkpoint size is a few kilobytes regardless of problem size.


Creating a Checkpoint
---------------------

While Mace4 is running, send it the SIGUSR2 signal:

  kill -USR2 <pid>

where <pid> is the Mace4 process ID.

To find the process ID:

  ps aux | grep mace4

Or, if you started Mace4 in the background:

  mace4 -f input.in > output.out 2> errors.err &
  echo $!          # prints the PID

Example:

  $ mace4 -n 10 -N 20 -f problem.in > run.out 2> run.err &
  [1] 12345
  $ kill -USR2 12345

Mace4 will write a checkpoint directory in the current working
directory and print a confirmation to stderr:

  Checkpoint written: mace4_12345_ckpt_50000 (depth=25, selections=50000)

The directory name includes the process ID and selection count:

  mace4_<pid>_ckpt_<selections>/

If checkpoint_exit is set, Mace4 exits with code 107. Otherwise, it
continues searching.


What the Checkpoint Contains
----------------------------

The checkpoint directory contains these files:

  metadata.txt       Domain size, model count, search depth, stats
  trail.txt          Decision trail (one line per depth level)
  saved_input.txt    Self-contained LADR input (options + clauses)
  options.txt        Human-readable option dump (not used on resume)

The trail.txt file has one line per search depth:

  <depth> <cell_id> <value_index> <last> <max_constrained>

Example:

  0 25 0 1 0
  1 26 0 2 1
  2 30 1 2 1

This says: at depth 0, cell 25 is trying value 0 (of 0..1),
max_constrained=0. At depth 1, cell 26 is trying value 0 (of 0..2),
max_constrained=1. And so on.

Depths 0 through N-2 are assignments currently in effect. Depth N-1
is the position where search continues on resume.


Resuming from a Checkpoint
--------------------------

To resume, use the -r flag with the checkpoint directory:

  mace4 -r mace4_12345_ckpt_50000

The -r flag is all you need. The saved_input.txt inside the checkpoint
directory contains the complete input (options and clauses), so you do
NOT need to supply the original input file again.

You can override options on the command line. For example, to give the
resumed run more time or change the end domain size:

  mace4 -r mace4_12345_ckpt_50000 -t 7200 -N 30

On resume, Mace4 prints restoration messages to stderr:

  Resuming from checkpoint: mace4_12345_ckpt_50000
  Resume: domain_size=15, total_models=194972, depth=197, selections=390130

  === Mace4 starting on domain size 15. ===
  Trail replay complete: 197 depths replayed

The search then continues exactly where it left off. The total model
count includes models found before the checkpoint.

IMPORTANT: If the checkpoint was created from a TPTP-mode run (e.g.,
mace4 -f problem.p), you do NOT need to pass -f problem.p on resume.
The checkpoint is self-contained and automatically detects that the
original run was TPTP mode.


Complete Example
----------------

1. Start a long-running model search:

   $ mace4 -n 10 -N 20 -t 86400 -f hard_problem.p > run1.out 2> run1.err &
   [1] 54321

2. After some time, create a checkpoint:

   $ kill -USR2 54321

   Mace4 saves the checkpoint and continues running.

3. If the machine needs to restart, or you want to stop:

   $ kill 54321

4. Later, resume from the checkpoint:

   $ mace4 -r mace4_54321_ckpt_500000 -t 86400 > run2.out 2> run2.err &

5. You can checkpoint the resumed run too:

   $ kill -USR2 $!

   This creates a new checkpoint directory (e.g., mace4_54322_ckpt_750000).


Checkpoint-and-Exit Workflow
----------------------------

If you want Mace4 to save and stop (not continue running), add this
to your input file:

  set(checkpoint_exit).

Then:

  $ mace4 -n 10 -N 20 -f problem.in > run.out 2> run.err &
  $ kill -USR2 $!

Mace4 writes the checkpoint and exits with code 107. You can check
the exit code:

  $ echo $?
  107

This is useful in scripts:

  #!/bin/bash
  mace4 -n 10 -N 20 < "$1" > run.out 2> run.err &
  PID=$!
  sleep 3600                    # run for 1 hour
  kill -USR2 $PID               # request checkpoint
  wait $PID
  EXIT=$?
  if [ $EXIT -eq 107 ]; then
      echo "Checkpointed successfully"
  elif [ $EXIT -eq 0 ]; then
      echo "Model found before checkpoint"
  else
      echo "Exited with code $EXIT"
  fi


Multiple Checkpoints
--------------------

You can send SIGUSR2 multiple times during a run. Each checkpoint
creates a new directory with the current selection count in its name:

  mace4_12345_ckpt_50000/
  mace4_12345_ckpt_150000/
  mace4_12345_ckpt_280000/

Earlier checkpoints are not overwritten. You can resume from any of
them.


Periodic Automatic Checkpoints
------------------------------

For long-running searches, you can have Mace4 create checkpoints
automatically at regular intervals:

  assign(checkpoint_minutes, 30).   % checkpoint every 30 minutes

This uses wall-clock time (not CPU time). The timer starts when the
search phase begins, so the first checkpoint appears N minutes into
the search (not counting setup time). Checkpoints are checked once per
cell selection in the DPLL loop.

The minimum interval is 15 minutes. If you set a value between 1 and
14, Mace4 uses 15 minutes.

By default, only the most recent 3 automatic checkpoint directories
are kept on disk. When a new periodic checkpoint is written and would
exceed this limit, the oldest automatic checkpoint is removed:

  assign(checkpoint_keep, 5).   % keep last 5 auto checkpoints

SIGUSR2 manual checkpoints are completely independent of rotation.
They are not counted against the checkpoint_keep limit and are never
removed by the rotation mechanism. You can freely mix manual and
automatic checkpoints.

Example: automatic checkpoints with manual override:

  % In LADR input file:
  assign(checkpoint_minutes, 30).
  assign(checkpoint_keep, 3).
  formulas(sos).
  f(f(x,y),f(x,f(x,y))) = x.
  end_of_list.

  $ mace4 -n 10 -N 20 < problem.in > run.out 2> run.err &
  [1] 54196

  % Mace4 creates checkpoints automatically every 30 minutes.
  % At any time, you can also manually checkpoint:
  $ kill -USR2 54196

  % The manual checkpoint is kept forever; automatic ones rotate.

You can resume from any checkpoint directory, whether it was created
automatically or manually. The -r flag works the same either way.


LADR Mode vs TPTP Mode
-----------------------

Checkpointing works identically in both modes. The key difference is
on resume:

  LADR mode:  saved_input.txt is read directly via the LADR parser.

  TPTP mode:  saved_input.txt is still in LADR format (the TPTP
              formulas were clausified and saved as LADR clauses).
              On resume, Mace4 detects tptp_mode=1 in metadata.txt
              and sets up the symbol table to match the original TPTP
              configuration before reading saved_input.txt.

You do not need to do anything special -- just use -r and Mace4
handles the mode detection automatically.


Model Count Accumulation
------------------------

The total model count is maintained across checkpoint/resume
boundaries. For example:

  Original run:  finds 194,972 models before checkpoint
  Resumed run:   finds 327,719 more models in 5 seconds

  Total reported: "Exiting with 522691 models."

The count in the "Exiting with" message is the cumulative total,
not just the models found after resume.


Options Reference
-----------------

  set(checkpoint_exit).         Exit (code 107) after writing checkpoint.
                                Default: off.

  clear(checkpoint_exit).       Continue searching after checkpoint.

  assign(checkpoint_minutes, N).  Periodic checkpoint interval in
                                minutes (wall-clock time).
                                Default: -1 (disabled).
                                Minimum: 15.

  assign(checkpoint_keep, K).   Max automatic checkpoint directories to
                                retain on disk.  Default: 3.  Range: 1-100.
                                Oldest auto checkpoint is removed when a
                                new one would exceed this count.  Does not
                                affect SIGUSR2 manual checkpoints.


Command-Line Reference
----------------------

  -r dir     Resume from checkpoint directory.

This is the only checkpoint-related command-line option. All other
checkpoint settings are controlled via the input file (or the options
saved in the checkpoint's saved_input.txt).


Exit Codes
----------

  0     max_models reached (model found)
  2     Search exhausted (all domain sizes tried)
  3     All models found (max_models = -1, search exhausted)
  4     max_seconds exceeded (models found)
  5     max_seconds exceeded (no models found)
  6     max_megs exceeded (models found)
  7     max_megs exceeded (no models found)
  101   Interrupted (SIGINT / Ctrl-C)
  107   Checkpoint saved (with checkpoint_exit set)


Differences from Prover9 Checkpointing
---------------------------------------

Feature                  Prover9                    Mace4
-----------------------  -------------------------  -------------------------
Search algorithm         Given-clause loop          DPLL backtracking
What is saved            Clause lists, indexes,     Decision trail only
                         justifications, formulas   (a few KB)
Checkpoint size          ~15% of RAM usage          A few kilobytes
Resume mechanism         Load clauses + re-index    Rebuild + replay trail
checkpoint_ancestors     Yes (for proof history)    N/A (no proofs)
Determinism              Given/Kept counts match    Exact replay of trail
Input on resume          Must supply original -f    Self-contained (-r only)
Directory naming         prover9_<pid>_ckpt_<given> mace4_<pid>_ckpt_<sel>


Troubleshooting
---------------

Problem: "assign_and_propagate: contradictory assignment" during resume
  Cause: The input or options changed between checkpoint and resume,
  causing the ground clause structure to differ. The saved decision
  trail references cell IDs from the original run that no longer exist
  or have different meanings.
  Fix: Do not modify the input between checkpoint and resume. Use -r
  alone (without changing the input file). Command-line overrides like
  -t (time) and -N (end size) are safe because they don't affect the
  cell table.

Problem: Checkpoint directory not created after SIGUSR2
  Cause: Mace4 checks for the signal between cell selections in the
  DPLL loop. If the search is between domain sizes or hasn't started
  yet, the signal may not be processed immediately.
  Fix: Wait a moment and send SIGUSR2 again. Mace4 also checks for
  pending checkpoints between domain sizes.

Problem: "cannot open .../saved_input.txt for resume"
  Cause: The checkpoint directory path is incorrect or the directory
  was moved/deleted.
  Fix: Verify the -r argument points to a valid checkpoint directory
  containing saved_input.txt.

Problem: Resumed run finds fewer models than expected
  The model count on the "Exiting with" line is cumulative. If you
  set a shorter time limit on resume, fewer additional models will be
  found, but the total includes all models from before the checkpoint.

Problem: Variables appear as integers in saved_input.txt
  This is expected. Mace4 uses integer-style variables internally
  during search. The saved_input.txt is written with standard variable
  names (x, y, z, ...) which are correctly parsed on resume.
